### 2. 商業背景

在現代協同教學（Collaborative Learning）的課堂情境中，教師面臨以下痛點：

- 傳統 Google Docs 等工具缺乏 Markdown 格式原生支援，不適合理工或技術性課程使用
- 多組學生使用同一份文件時，版本管理混亂
- 學生作業繳交往往是散亂的文字檔或截圖，缺乏持久化、可追溯的記錄機制
- 現有協作工具的財務成本對教育機構而言偏高

### 3. 商業目標

| 目標 | 說明 |
|---|---|
| 降低協作工具成本 | 以自建開源方案取代商業 SaaS |
| 提升課堂參與追蹤 | 透過 GitHub 存檔，保留每次課堂的歷史紀錄 |
| 強化學習成果可視化 | 多組學生的文件集中存放於同一 Repo，方便統一批閱 |
| 可擴充性 | 支援每年新增班級，不需重架系統 |

### 4. 利害關係人

| 角色 | 需求摘要 |
|---|---|
| 授課教師 | 能夠分組、部署、追蹤學生產出 |
| 學生 | 能夠即時協作編輯，無需安裝任何軟體 |
| 學校 IT 管理員 | 系統好安裝、好維護，資安風險低 |

### 5. 成功指標（KPI）

- 每堂課至少 1 次 GitHub 存檔（Commit）
- 同組 2 人以上可同時編輯，延遲 < 500ms
- 系統上線後，教師設定時間 < 15 分鐘

# 二、MRD — 市場需求文件
## Market Requirements Document

### 1. 目標市場

**主要市場**：台灣高中、大學的資訊、技術性學科課堂（約 15-40 人/班）

**次要市場**：
- 程式設計訓練營（Bootcamp）
- 企業內部技術文件協作小組

### 2. 使用者輪廓（Persona）

#### Persona A — 林老師（授課教師，39 歲）
- 教授「資訊科技」或「計算機概論」
- 希望學生在課堂上即時合作寫報告
- 技術能力：熟悉基本 Git 操作，有 GitHub 帳號
- 痛點：用 Google Docs 管理多組文件很雜亂；Hackmd 沒有學生帳號識別

#### Persona B — 阿志（學生，17 歲）
- 第一次接觸 Markdown
- 只需要瀏覽器，不需要安裝軟體
- 希望打字就能跟同學即時同步，不用 email 傳檔案

### 3. 競品分析

| 競品 | 優點 | 缺點 |
|---|---|---|
| HackMD | Markdown 友善，有共享模式 | 無法識別每位編輯者身份，免費版有限制 |
| Google Docs | 普及、穩定 | 不支援 Markdown，格式易跑版 |
| Notion | 功能豐富 | 學習曲線高，免費版有頁面上限 |
| **Markdown-Collab** | 免費自建、Markdown 原生、身份識別、自動 GitHub 歸檔 | 需自行部署伺服器 |

### 4. 市場差異化優勢

- **身份識別**：以 GitHub OAuth 辨識每位學生，游標顯示真實頭像與暱稱
- **自動歸檔**：每次儲存即 Commit 至 GitHub，天然保存歷史版本
- **零安裝**：學生只需瀏覽器，用網址即可加入指定組別
- **多組隔離**：一套系統支援無限組別，透過 URL 參數分組

# 三、PRD — 產品需求文件
## Product Requirements Document

### 1. 產品概覽

**產品名稱**：Markdown-Collab

**產品類型**：自建式多人即時 Markdown 協作編輯器

**核心技術棧**：React 19 + Milkdown v7 + Yjs + Supabase Auth + GitHub API

---

### 2. 功能需求

#### F-01：動態協作房間
| 欄位 | 說明 |
|---|---|
| 描述 | 透過網址參數 `?room=` 動態分配 Yjs 協作房間 |
| 輸入 | `http://localhost:5173/?room=2026/classA-group1` |
| 輸出 | 該組學生進入獨立的即時協作空間 |
| 優先級 | P0（核心功能） |

#### F-02：GitHub OAuth 身份識別
| 欄位 | 說明 |
|---|---|
| 描述 | 學生以 GitHub 帳號登入，游標顯示頭像與暱稱 |
| 前置條件 | Supabase 專案已啟用 GitHub Auth Provider |
| 優先級 | P0 |

#### F-03：即時協作同步
| 欄位 | 說明 |
|---|---|
| 描述 | 多人同時編輯，字元級別即時同步，使用 Yjs CRDT |
| 延遲要求 | 本地網路環境下 < 100ms |
| 優先級 | P0 |

#### F-04：GitHub 存檔
| 欄位 | 說明 |
|---|---|
| 描述 | 點擊按鈕後，將目前 Markdown 內容透過 GitHub API 存至指定路徑 |
| 存檔路徑 | `docs/{room}.md`（支援斜線，自動建立子資料夾） |
| 安全機制 | GitHub PAT 儲存於伺服器端，學生無法取得 |
| 優先級 | P0 |

#### F-05：表格操作工具列
| 欄位 | 說明 |
|---|---|
| 描述 | 游標進入表格時自動顯示工具列：插入列/欄、刪除列/欄/整表 |
| 優先級 | P1 |

---

### 3. 非功能需求

| 需求 | 說明 |
|---|---|
| 可用性 | 系統啟動指令 ≤ 2 個（`node ws-server.mjs` + `npm run dev`） |
| 安全性 | GitHub PAT 不暴露於前端，由 `ws-server.mjs` 或 Supabase Edge Function 代理 |
| 擴充性 | 新增班級/組別無需修改任何程式碼，僅需調整網址參數 |
| 相容性 | 支援 Chrome / Edge / Firefox 等主流瀏覽器，學生端無需安裝 |
